如何干上痛苦的工作

再读《穷查理宝典》，至查理演讲谈及如何过上痛苦的人生，打乒乓时思绪漫游，联想到这个问题，梳理下自认为的处方

  1. 去和价值观不同的一起工作，不去寻找加入优秀团队的机会
  2. 去做没有价值的事，沉溺于琐碎小事，应付交待的事，不去挖掘事项价值
  3. 去抱怨工作环境，企图靠换坑去逃避现状，不去发掘问题的本质
  4. 不做授之以渔之事，”带教”放养或不去启发思考，不培养有潜力的年轻人去承担工作，也不想着去承担更有价值更复杂的工作
  5. 不做技术沉淀，不留心产出，只做重复性工作，或者遇到任务照搬经验，快速闭环大吉
  6. 不做技术分享，不去强化能力知识的掌握程度，不去谋求清晰的表达能力
  7. 不做份内之外的事，不去试图从工作获取成就感
  8. 不去关心组内经验不够丰富同事日常工作进展，不去了解收集他们所遇难点痛点，不去给出解决方案建议
  9. 不做到今日重要事项今日毕，把事情拖到明天，拖到项目延期
  10. 不去推进上游下游瓶颈优化，不去关注研发提测质量，承接压力到团队
  11. 不在会议发表有建设性意见，私下吐槽考虑不周全或坐等看笑话
  12. 不去跟进经验不够丰富同事工作任务质量，泄露问题时对其进行责问
  13. 不去及时分享掌握的差异信息，保持信息不对称的优异感
  14. 不去关注产品的趋势和竞品差异，只做平庸的产品
  15. 不去关注产品线上问题，不关注业务外围满意度
  16. 不做向上管理，只吭哧吭哧埋头干活，企图让上级凭空洞察想法
  17. 不去做交付上线验证清单，交付过程问题推给研发

如何过上痛苦的人生

其实早在三四年前在26厂工作时，应团队内组织的分享，当时正好在读这本书，索性就选择了这个主题。可惜当时没有将演讲稿记录博文，没法得知当时是怎么思考，亦无法对照当时自己给的处方，甚是可惜。依稀记得当时分享一塌糊涂，屡屡忘词，哼哈词无数。。。

付上查理提及的几个过上痛苦人生处方（前两个是约翰尼卡森提的）

  • 为了改变心情或者感觉而使用化学物质
  • 妒忌以及怨恨
  • 要反复无常，不要虔诚地做你正在做的事
  • 尽可能从你们自身的经验获得知识，尽量别从其他人失败或成功的经验中广泛吸取教训
  • 当在人生的战场上遭遇第一、第二或者第三次严重失败时，就请意志消沉，从此一蹶不振
  • 请忽略那个乡下人故事，曾经有个乡下人说“要是知道我会死在哪里就好了，那我将永远不去那个地方”

虚荣的我

什么时候兴趣爱好成了满足虚荣心（刻意追求别人眼里的“高大上”）

今日沿江宁东南一路骑行，途遇有钓鱼佬及停在路边的电瓶车，是追求内心的平静，是忙碌工作外的调剂，亦或是纷扰家庭的解脱，我不得而知；有熙熙攘攘穿行不息的货车；有年久失修的房屋，零星住户飞檐悬挂的细细光纤已藤蔓缠绕；有采石场荒废留下近乎90度的绝壁山体；有年愈七十老妇烈日下运送废品；有一群尚不退休大爷大妈们舞起镰刀为城市绿化发挥余热；

惭愧，单凭一纸文凭和靠几年995/6时间堆积而来的工作经验，肆意的蹉跎人生，事业毫无追求仅逐利而去，生活更是浑噩依旧，仅有的一些想法和技能提升，亦是围绕职场逐利

回首人生

  • 童年 无拘无束，略显聪慧；探索自然，游心乡野
  • 县中 陷入题海，尚游刃有余；在意师长赞许肯定，排名优先
  • 大学 沉迷游戏，应付学业；除了面包和水，虚拟世界什么都有
  • 国企生涯 延续在校时光；游山涉水，贪婪追求丰富感官体验
  • 切换设备商 疲于赚钱，恶补工作技能，游走在996边缘的小白；省心收心，远足热情淡却
  • 回归互联网 逐利而行，积累经验；追求丰富阅读以体验不同的人生和提升认知

及至当下，工作方面认定自己是平庸之人，在自我认为遭遇不公时，不去且不愿做有意义的改变，一再逃避，靠推卸责任指责他人去敷衍麻痹自己（联想到舔，实属荒谬想法，领导用人以长，他人长处真就视而不见？）；工作执行中所遇困难，因并非创造性工作，且历年经验积累尚游刃有余，认为办法总比困难多（即NLP能力层次），习惯于舒适区做决策开展事务，不愿主动去发掘或插手有挑战性的事情

大胆探索，郑重做事

时至今日，依旧没有找到什么感兴趣的事。分析原因或许有二，其一，探索不够深不够广，深度不够，易尚未发现乐趣便半途而废，广度不足，沉溺于当下舒适安逸氛围，束缚欲望，局促于柴米油盐酱醋茶，蹉跎度日；其二，便是做事态度不够郑重

梁漱溟《朝话》提及“自己有短处，有为难处，也要照样子摆出来。若力量没尽到而搪塞掩饰是虚伪，若力量没尽到而把懒惰摆出来给人看便是无耻”

故接下来，以郑重的态度做事，真诚沟通，对于那些与自己价值观有差异，或是自认为没有意义的事，更需要真诚沟通，了解他人想法，是否是自身没意识到的；一旦承诺便郑重去做，而不是功利性的选择去做，更不是敷衍实施隐患无穷

经验之谈

总结

  • 用到就学，止步完成
    就我自身而言，工作这么多年，算是一个反面例子，首先没有去钻研具体的技术，基本上就是用到就学的方式，是一种工作压力下的被动学习，其实当解决问题也就终止了学习，没有去将掌握的知识做相关的沉淀
    所以开始杨哥让我分享，其实我心里也没底，感觉可以输出些东西，又同时觉得这些东西并不重要
    人与人之间的差距不是完成一件事，而是完成这件事后沉淀的部分；完成一件事就终止学习及复盘，那么下次遇到同样的事情还好，经验的帮忙下可以快速解决，不同事项又要从头开始；而做沉淀和复盘，理想的可能性是相关性大的事项任务来临时，能够有应对方案
    （有个很明显的例子，比如上次巴西站三方索引字段比对，王总提了个建议，将索引内容用http请求调用返回response，可以复用现有成熟工具棱镜diff功能验证，这个一直都没有实施，比对的脚本能力也就终止于当前的json比对，失去了本来可以输出或沉淀通过价值机会）
    站在更高的角度
  • 探索式 学习vs工作
    学习探索式可以快速掌握陌生知识；工作任务推进用探索式就比较鲁莽，也容易走弯路；如何才能在任务执行时少用探索式，依赖于平日的沉淀积累
  • 速度与速率
    物理学矢量与标量
    方向及选择的重要性，投入工作技能提升时间是有限的，如何才能发挥更大的影响力，
  • 复利效应
    次数和利率
    做工作只想着不拖延完成任务，从未过多去思考为什么做这件事，往往主次不分关键处未得以充分实施；遇一事则急于照搬套路解决，事情完成了便闭环大吉了，陷入低水平勤奋陷进（引入复利效应来说，虽然执行次数足够多，但利率微乎其微，故难以有指数型增长）。归根结底太浮躁，太急于反馈，无法静下心来做事，挖掘事情背后的本质，解决什么问题，为什么要解决这个问题，还有没有其他更好的方式解决这个问题

软件测试面试题V1.0

因最近团队发展需要招聘测试人员，现梳理下软件测试相关基础面试题

开场

  • 你好，很高兴认识你，我是今天面试官xx，麻烦用1-2分钟的时间做个简单的自我介绍

shell

  • 如何快速生成1个200w行的测试文件
    for i in {0..2000000};do echo “XXX”>>test.csv;done
  • 切换用户
    su
  • 拷贝远程文件到本地
    scp root@hostA:/path/filename /var/www/local_dir
  • 取nginx进程PID进程号
    ps -aux |grep nginx |grep -v grep |awk ‘{print $2}’
  • 查看修改定时任务
    crontab -l

sql

  • 一个retry表，表里有一个主键，还有一列column是失败重试次数叫cnt，值是0~5，要你输出每种cnt的数量，这个应该怎么写
    select cnt,count(*) from retry group by cnt;
  • mysql去重用什么关键字
    distinct
  • 给student表加个字段语句
    alter table stud add column age int;
  • ddl和dml了解吗？
    DML(Data Manipulation Language)数据操纵语言：适用范围：对数据库中的数据进行一些简单操作，如insert,delete,update,select等.
    DDL(Data Definition Language)数据定义语言：适用范围：对数据库中的某些对象(例如，database,table)进行管理，如Create,Alter和Drop

web

  • 接口测试，就http接口而言，测试中会关注哪些？
    接口入参param，请求头headers，请求body，出参，并发性能，安全性等
  • http请求方法有哪些
    GET、POST、HEAD、PUT、DELETE、OPTIONS、TRACE、CONNECT

编程语言

  • 复制列表，列表删除最后一个元素，列表删除指定元素？
  • 有两个json（或字典）格式数据，获取差异部分，这个怎么实现？
  • 实现下斐波那契递归函数
  • 迭代器、生成器了解吗？有什么简单的用法
  • 你有用过哪些第三方库
  • 写过哪些脚本，举个例子，脚本实现了什么功能？

项目

  • 说说你的项目软件测试流程？
  • 简单介绍下你的项目业务框架或整体的架构流程
  • 你负责的测试功能模块，具体是怎么测试的？
  • 项目测试过程有遇到什么问题吗，怎么解决的？
  • 这个项目迭代周期多长时间，开发和测试分别占了多少时间？
  • 项目中有没有让你印象特别深刻的bug？
  • 遇到bug怎么定位是前端问题还是服务端问题？
  • 在项目测试过程中，如果开发说你发现的一个问题不是bug，这个时候你是怎么解决的？
  • 在这个项目中，你觉得还有什么地方可以优化的吗？
  • 在定位问题的过程中，你会采取什么方式？或者说，怎么分析和定位bug？
    日志、库表、代码

测试思维

  • 工作中有涉及测试方案和用例设计吗？
  • 测试案例设计有用到什么方法吗？

性能&自动化测试

  • 说说你们的自动化建设，使用了什么自动化框架，自动化框架是如何搭建的？
  • 做了哪些性能测试，有用到哪些工具，性能测试的关注点，有基准数据吗？
  • 有做过性能调优吗？

结束

浅谈平均速度与平均速率

Posted on 2022-05-17 Edited on 2022-05-18 Word count in article: Reading time ≈ NaN:aN

平均速度与平均速率

心血来潮，看了Walter Lewin的《经典力学》，Lesson1中量纲分析和误差醍醐灌顶，此处撇开不谈
因课程有提到平均速度和平均速率，结合当前的工作及带教，有一些想法迸发而来，随简单谈谈

  • 首先，回顾下物理知识，平均速度=位移/时间，平均速率=距离/时间；位移是矢量，距离为标量；故两者最大区别所在即是“方向”
  • 对应到职业生涯实际工作来说，速率很好理解，用工作年限内完成的项目任务等等去除以这段时间；速度也很好理解，换工作时只有跟求职要求相关的才是有效工作项目经验，加分项
  • 工作中，有时充实感往往来源某段时间内工作任务完成进度，抱着功不唐捐的想法，偶尔加加班忍不住发个朋友圈自我感动一下；其实归根结底这些都是“距离”层次的积累，在没有合理明确职业发展方向的前提时，所做的事只能算作“苦劳”
  • 联想到把价值输出放首位的自己，是否清晰的认识到距离和位移的区别，没有明确方向夸夸而谈的输出只能是“距离”量的积累，只有真正有价值的那部分，才是有效的，也是自己真正提升的“位移”，即所谓的“质”；量变到质变纯属侥幸，唯有先聚焦于质后持之以恒才终有“质变”
  • 对于带教的新人来说，也正是如此，不应该仅关注于最近一段时间完成了几个项目任务，完成了哪些脚本工具，学习掌握了多少知识；而应该把从项目任务完成中收获了什么，脚本工具新知识运用到工作中实际提效的部分，还有哪些场景可以复用，工作方面的学习充电是不是职业岗位方向当前阶段所需要的，这些放在首位；时刻牢记位移与距离本质区别，将有限精力从无用功处剥离

加速度与潜力&高负荷

物理所说的加速度，有一个很明显的例子，就是再说汽车的爆发力跑车的性能时会提到几秒钟加速到xx速度之类的，对应到生活上，个人觉得可以对应到一个名词，即潜力

  • 对于一个职场人，尤其是新人来说，领导同事一个关注点是新人的潜力怎么样，掌握某方面技能熟练度的效率几何；谈及潜力效率，就可以关联上物理学中的“加速度”，如何在短时间内迅速掌握业务流程&工作技能尤为重要
  • 区别于某些体制内相对稳定闲散的工作，互联网的工作往往会有版本交付的压力，某段时间会迎来高负荷的工作场景，这时候工作内容夹杂着部分有困难有挑战的部分，就需要有快速掌握新技术能力的要求，考量到个人身上，也就是考量自己的职场学习/应用/管理/协调等各方面能力“加速度”究竟几何
  • 潜力弱能提升吗？很明显是可以的，拿打羽毛球来说，其中很重要的一个是反应能力，这个可以通过其他运动如打篮球等提升；类比到工作中，认真打磨一门编程语言，在新项目需要切换其他语言时，这种底层相同的高相似场景，切换起来并不会太难，应为真正底层的本质是相通的

总结

纵观上述，其实不难理解，那些在职场职业生涯岗位上做着有效位移，磨炼加速度的人，往往可以成为骨干核心员工，进而走上管理岗位或成为技术专家
